## 3. Dataset Construction

### 3.1 Overview

IRC-Bench is constructed through a four-stage automated pipeline that transforms oral history transcripts into implicit entity recognition samples. Each sample consists of a first-person narrative that references a named entity through contextual cues alone, without ever naming it. The pipeline leverages GPT-4.1-mini for entity extraction, summary generation, and implicit rewriting, producing 25,136 benchmark samples spanning 12,337 unique entities.

### 3.2 Source Collections

The raw data comprises 1,994 cleaned oral history transcripts drawn from 11 thematic collections:

| Collection | Transcripts | Description |
|---|---:|---|
| Veterans | 517 | Military service narratives |
| Immigration | 402 | Immigration and assimilation experiences |
| Regional | 314 | Regional and community histories |
| Depression Era | 213 | Great Depression oral histories |
| Japanese American | 156 | Japanese American internment and post-war experiences |
| Academic | 153 | Academic and university histories |
| September 11 | 72 | 9/11 experiences and aftermath |
| Civil Rights | 68 | Civil rights movement narratives |
| COVID-19 | 42 | Pandemic experiences |
| Labor | 30 | Labor movement histories |
| Refugee | 27 | Refugee experiences |
| **Total** | **1,994** | |

These collections provide broad topical diversity, covering military conflicts, social movements, immigration, public health crises, labor history, and academic life. The first-person narrative style of oral histories naturally provides rich contextual cues (dates, locations, relationships, roles, events) that make implicit entity references solvable for knowledgeable readers, while remaining challenging for automated systems.

### 3.3 Pipeline Stages

**Stage 1: Transcript Cleaning.** Raw oral history transcripts are cleaned and converted to structured YAML format, preserving the first-person narrative voice while removing interviewer questions and metadata artifacts.

**Stage 2: Named Entity Recognition.** GPT-4.1-mini performs NER on each transcript, identifying named entities of seven types: Place, Organization, Person, Event, Work, Military Unit, and Other. Each extracted entity is linked to Wikidata (QID) and Wikipedia where possible. This stage produces 31,284 entity mentions across 1,752 transcript files (87.9% coverage).

**Stage 3: Explicit Summary Generation.** For each (transcript, entity) pair, GPT generates a first-person narrative summary focused on that entity, preserving the contextual cues surrounding the entity's mention. This produces 25,161 explicit summaries from 1,601 transcript files (80.3% coverage).

**Stage 4: Implicit Rewriting.** Each explicit summary is rewritten by GPT to remove all direct mentions of the entity name while preserving all contextual cues. The entity reference is replaced with generic descriptions (e.g., "Attack on Pearl Harbor" becomes "the attack on the naval base in Hawaii"). This produces the final 25,136 implicit rewrites (80.2% coverage), which form the benchmark's `implicit_text` field. The slight reduction from Stage 3 reflects a small number of cases where the entity could not be satisfactorily anonymized.

### 3.4 Entity Knowledge Base

The entity knowledge base contains 12,337 unique entities with the following metadata coverage:

- **Wikipedia URLs**: 84.6% of entities have associated Wikipedia pages
- **LLM-generated descriptions**: 70.9% have entity descriptions generated by GPT
- **Aliases**: 51.2% have alternative names sourced from Wikidata

Entity representations in the knowledge base serve multiple roles: as retrieval targets for closed-world experiments, as alias sources for evaluation matching, and as description inputs for embedding-based approaches.

### 3.5 Entity-Level Train/Dev/Test Splitting

To ensure rigorous evaluation, the dataset is split at the entity level rather than the sample level. All samples for a given entity appear in exactly one partition. This prevents information leakage where a model might learn entity-specific patterns from training examples and exploit them at test time.

The split uses a 70/10/20 ratio (seed=42), producing:

| Partition | Samples | Entities |
|---|---:|---:|
| Train | 17,971 | 8,635 |
| Dev | 2,532 | 1,234 |
| Test | 4,633 | 2,468 |
| **Total** | **25,136** | **12,337** |

Entity overlap between partitions is strictly zero: no entity appears in more than one partition.

### 3.6 Entity Type Distribution

The dataset exhibits a natural long-tail distribution over entity types:

| Type | Samples | Percentage |

## 4. Methodology

### 4.1 Task Formulation

**Implicit Entity Recognition (IER)** is defined as follows: given a first-person narrative text $t$ that implicitly references a named entity $e$ without ever mentioning $e$ by name, the task is to identify $e$. The text $t$ contains contextual cues (dates, locations, events, people, roles, descriptions) that jointly constrain the identity of $e$, but the model must synthesize these cues and draw on world knowledge to produce the correct entity name.

We evaluate IER under two formulations:

**Open-world formulation.** The model generates the entity name as free-form text, without access to a candidate set. This tests the model's ability to recall entities from its parametric knowledge. The open-world setting is more realistic, as it does not assume a closed inventory of possible entities.

**Closed-world formulation.** The model ranks all 12,337 entities in the knowledge base by relevance to the query text, selecting the highest-ranked candidate. This tests the model's ability to match implicit descriptions to entity representations via embedding similarity. The closed-world setting provides Hit@K metrics and is analogous to entity linking with a fixed knowledge base.

### 4.2 Open-World Methods

#### 4.2.1 LLM Generative Approach

We evaluate LLMs in a generative setting where each model receives the implicit text and produces the entity name. All models use temperature 0.0 (greedy decoding) and a maximum of 100 output tokens.

**System prompt** (shared across zero-shot and few-shot):

```
You are an entity recognition expert. Given a text that implicitly
references a named entity without mentioning it, identify what entity
is being referenced.
```

**Zero-shot prompt:**

```
What named entity is implicitly referenced in this text? The entity
is never mentioned by name.

Text: "{text}"

Think about the contextual cues (dates, places, events, people, roles)
and identify the specific named entity being referenced.

Answer with ONLY the entity name (canonical Wikipedia name), nothing else.
```

**Few-shot prompt** (5 fixed demonstrations):

```
What named entity is implicitly referenced in this text? The entity
is never mentioned by name.

Examples:
Text: "I remember that Sunday morning in December '41. We were listening
to the radio when the news broke about the attack on the naval base in
Hawaii. That's when everything changed."
Entity: Attack on Pearl Harbor

Text: "I enlisted right out of high school and went to boot camp in
San Diego. As an aircraft mechanic, I was sent to the Pacific."
Entity: United States Marine Corps

Text: "After the surrender, we flew into the main islands. I landed in
the bay and spent six months there for the occupation. The capital was
flattened by the B-29s."
Entity: Tokyo

Text: "In late 1941, I was set to ship out from San Francisco. A friend
ran up saying they're bombing the base in Hawaii."
Entity: Attack on Pearl Harbor

Text: "Growing up in that bustling metropolis with towering skyscrapers,
I was immersed in a vibrant culture."
Entity: New York City

Now identify the entity in this text:
Text: "{text}"

Answer with ONLY the entity name (canonical Wikipedia name), nothing else.
```

### 4.3 Closed-World Methods

#### 4.3.1 Embedding Retrieval Approach

In the closed-world setting, we encode both the implicit query text and all 12,337 entity representations into a shared embedding space, then rank entities by cosine similarity. We explore three entity representation strategies:

- **Name**: the entity name alone (e.g., "University of California, Berkeley")
- **Description**: the entity name concatenated with its LLM-generated description (e.g., "University of California, Berkeley: A public research university in Berkeley, California...")
- **Wiki**: the first sentence from the entity's Wikipedia article

For entities lacking a description or Wikipedia text, we fall back to the next available representation (Wiki falls back to Description, which falls back to Name).

#### 4.3.2 BGE-base Baseline (C1, C2, C3)

We use BAAI/bge-base-en-v1.5 as our baseline embedding model. This 110M-parameter model produces 768-dimensional embeddings and ranks among the top general-purpose bi-encoders on the MTEB benchmark. Embeddings are L2-normalized before computing cosine similarity.

| ID | Model | Entity Representation |
|---|---|---|
| C1 | BGE-base (off-the-shelf) | Name |
| C2 | BGE-base (off-the-shelf) | Description |
| C3 | BGE-base (off-the-shelf) | Wiki |

Encoding uses a batch size of 64 for both entity and query embeddings.

#### 4.3.3 DPR Fine-tuning (C4, C5, C6)

We fine-tune BGE-base using a Dense Passage Retrieval (DPR) approach with Multiple Negatives Ranking Loss (MNRL). Each training pair consists of an implicit text (query) and its gold entity representation (positive passage). MNRL uses in-batch negatives: for a batch of $B$ query-positive pairs, each positive for one query serves as a negative for all other queries, providing $B-1$ negatives per sample without explicit hard negative mining.

| ID | Model | Entity Representation |
|---|---|---|
| C4 | DPR fine-tuned | Name |
| C5 | DPR fine-tuned | Description |
| C6 | DPR fine-tuned | Wiki |

**Training configuration:**

- Base model: BAAI/bge-base-en-v1.5
- Training examples: 17,971 (full train split)
- Epochs: 3
- Batch size: 48
- Learning rate: 2e-5
- Warmup steps: 100
- Mixed precision: FP16 (AMP)
- Loss function: MultipleNegativesRankingLoss
- Optimizer: AdamW (default sentence-transformers settings)

The fine-tuned model is evaluated on the same test set of 4,633 samples against all 12,337 entities.

### 4.4 RAG Baseline (RAG1)

We implement a Retrieval-Augmented Generation (RAG) baseline that combines embedding retrieval with LLM reranking. The pipeline operates in two stages:

1. **Retrieval stage.** BGE-base with entity descriptions (C2 configuration) retrieves the top-5 candidate entities for each implicit query.
2. **Reranking stage.** GPT-4.1-mini receives the implicit text along with the 5 candidates (with their descriptions) and selects the most likely entity or suggests a better one.

**RAG prompt:**

```
This text implicitly references a named entity without naming it.
Based on the contextual cues, which candidate is most likely?

Text: "{text}"

Candidates:
1. {candidate_1} - {description_1}
2. {candidate_2} - {description_2}
3. {candidate_3} - {description_3}
4. {candidate_4} - {description_4}
5. {candidate_5} - {description_5}

If none match well, suggest a better entity.
Answer: [number]. [entity name]
```

This approach tests whether an LLM can effectively rerank retrieved candidates to improve over pure embedding retrieval.

### 4.5 Fine-tuned LLM (O10)

We fine-tune Llama 3.1 8B Instruct using QLoRA (Quantized Low-Rank Adaptation) for the IER task. The model is trained to generate the entity name given the implicit text, using the standard causal language modeling objective.

**Prompt template:**

```
<|begin_of_text|><|start_header_id|>system<|end_header_id|>
You identify implicitly referenced entities.<|eot_id|>
<|start_header_id|>user<|end_header_id|>
What entity is implicitly referenced? Answer with only the entity name.

Text: {implicit_text}<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>
{entity}<|eot_id|>
```

**Training configuration:**

- Base model: meta-llama/Llama-3.1-8B-Instruct
- Quantization: 4-bit NormalFloat (NF4) via BitsAndBytes
- Compute dtype: bfloat16
- LoRA rank ($r$): 16
- LoRA alpha ($\alpha$): 32
- LoRA dropout: 0.05
- Target modules: q_proj, v_proj, k_proj, o_proj
- Training batch size: 16
- Gradient accumulation steps: 2 (effective batch size: 32)
- Epochs: 2
- Learning rate: 2e-4
- Maximum sequence length: 192 tokens
- Warmup steps: 50
- Precision: bfloat16
- Evaluation batch size: 32
- Generation: greedy decoding, max 30 new tokens
- Training data: 17,971 samples (full train split)
- Validation data: 500 samples (subset of dev split)
- Framework: TRL SFTTrainer with PEFT

## 5. Evaluation Protocol

### 5.1 Matching Hierarchy

Entity names can be expressed in multiple valid forms (e.g., "United States Marine Corps" vs. "USMC" vs. "Marines"). To account for this variation, we employ a four-tier matching hierarchy, applied in order of decreasing strictness:

1. **Exact match (Tier 1):** The prediction and gold entity are identical after lowercasing and whitespace trimming.
2. **Alias match (Tier 2):** The prediction matches one of the gold entity's known aliases from Wikidata. For example, predicting "NYC" for gold entity "New York City" is an alias match.
3. **Containment match (Tier 3):** The prediction is a substring of the gold entity, or vice versa. For example, predicting "Pearl Harbor" for "Attack on Pearl Harbor" qualifies as a containment match.
4. **Jaccard match (Tier 4):** The token-level Jaccard similarity between the prediction and gold entity is at least 0.5. This captures partial overlaps where the prediction includes most of the relevant tokens.

A prediction is considered correct at a given tier if it matches at that tier or any stricter tier. When reporting alias-aware accuracy (the primary metric for open-world experiments), we count any prediction that achieves Tier 1 or Tier 2 as correct. When reporting the broader Hit@1 metric used in the alias evaluation, all four tiers contribute.

### 5.2 Metrics

**Open-world experiments:**

- **Exact match**: proportion of predictions that exactly match the gold entity name (Tier 1 only)
- **Alias match**: proportion of predictions that match via exact or alias matching (Tiers 1 and 2)
- **Containment match**: extends alias match with substring containment (Tiers 1, 2, and 3)
- **Jaccard match**: extends further with token-level Jaccard similarity (all four tiers)

**Closed-world experiments:**

- **Hit@K** (K = 1, 3, 5, 10): proportion of queries for which the gold entity appears in the top-K ranked candidates
- **MRR** (Mean Reciprocal Rank): average of $\frac{1}{\text{rank}}$ across all queries, where rank is the position of the gold entity in the ranked list

### 5.3 Entity-Level Splitting Guarantee

The entity-level split ensures zero entity overlap between partitions (train/dev/test). This is verified programmatically: the intersection of entity sets across all partition pairs is empty. This design choice prevents models from memorizing entity-specific patterns during training and ensures that test-time performance reflects genuine generalization to unseen entities.

### 5.4 Statistical Significance

To assess whether performance differences between methods are statistically significant, we use McNemar's test (with continuity correction) on the paired per-sample outcomes from each pair of compared systems. Additionally, we compute bootstrap confidence intervals (1,000 resamples, seed=42) at the 95% level to characterize the uncertainty around each system's accuracy estimate.

### 6.2 Key Findings

**Finding 1: GPT-4o few-shot achieves the best open-world performance.** O2 (GPT-4o, few-shot) achieves 31.62% exact match and 38.94% alias match, outperforming all other open-world configurations. The gap between exact and alias match (7.32 percentage points) indicates that a substantial fraction of GPT-4o's predictions are valid alternative names for the gold entity. This effect is especially pronounced for GPT models and suggests the importance of alias-aware evaluation for IER.

**Finding 2: Few-shot examples consistently improve performance.** Adding five fixed demonstrations improves GPT-4o from 27.02% to 31.62% exact match (+4.60 pp) and GPT-4.1-mini from 25.71% to 28.66% (+2.95 pp). For Llama 3.1 8B, the improvement is from 13.92% to 17.83% (+3.91 pp). All differences are statistically significant (McNemar's test, p < 0.001). The few-shot examples appear to calibrate the model's output format and entity granularity, reducing cases where models produce entity types instead of specific entity names.

**Finding 3: DPR fine-tuning dramatically improves closed-world retrieval.** Fine-tuning BGE-base with MNRL loss more than doubles Hit@1 for all entity representation types:
- Name: 16.51% to 30.00% (+13.49 pp)
- Description: 16.64% to 35.38% (+18.74 pp)
- Wiki: 14.38% to 27.95% (+13.57 pp)

The best closed-world configuration (C5: DPR + Description) achieves 35.38% Hit@1 and 71.49% Hit@10, indicating that the gold entity frequently appears in the top-10 candidates even when it is not ranked first.

**Finding 4: Entity descriptions are the most effective representation for retrieval.** Among both baseline (C1, C2, C3) and fine-tuned (C4, C5, C6) models, entity descriptions consistently outperform entity names and Wikipedia sentences. C5 (DPR + Description) outperforms C4 (DPR + Name) by 5.38 pp on Hit@1 and C6 (DPR + Wiki) by 7.43 pp. This suggests that LLM-generated descriptions capture more of the contextual information present in implicit texts than either short entity names or Wikipedia lead sentences.

**Finding 5: Chain-of-thought hurts performance on IER.** Contrary to expectations, CoT prompting degrades performance relative to direct prompting:
- GPT-4o: CoT (22.36%) vs. ZS (27.02%), a decline of 4.66 pp
- GPT-4.1-mini: CoT (18.95%) vs. ZS (25.71%), a decline of 6.76 pp
- Llama 3.1 8B: CoT (9.52%) vs. ZS (13.92%), a decline of 4.40 pp

This suggests that extended reasoning introduces opportunities for the model to "talk itself into" incorrect answers, particularly when the model fixates on a single cue and rationalizes a wrong entity. The CoT experiments also use temperature 0.7 (vs. 0.0 for direct prompting), which may contribute additional noise.

**Finding 6: Model scale matters substantially.** GPT-4o ZS (27.02%) outperforms Llama 3.1 8B ZS (13.92%) by 13.10 pp, and Llama 3.1 8B ZS outperforms Llama 3.2 1B ZS (Jaccard: 17.27%) by a smaller margin. The difference between GPT-4o and Llama 3.1 8B is statistically significant (McNemar's chi-squared = 432.28, p < 0.001), with 892 samples correct only by GPT-4o versus 203 correct only by Llama 8B.

**Finding 7: RAG underperforms direct LLM generation.** The RAG1 configuration (BGE retrieval + GPT-4.1-mini reranking) achieves 19.71% exact match, substantially below GPT-4.1-mini zero-shot (25.71%) and few-shot (28.66%). This indicates that the retrieval stage is a bottleneck: when the gold entity does not appear among the top-5 candidates (which occurs in roughly 67% of cases with BGE-base), the LLM reranker cannot recover it, despite having the option to "suggest a better entity."

**Finding 8: Open-world methods outperform closed-world baselines.** Comparing the best open-world result (O2: 41.10% Jaccard) against the best baseline closed-world result (C2: 16.64% Hit@1 raw; 22.58% alias-aware), open-world LLM generation substantially outperforms off-the-shelf embedding retrieval. However, fine-tuned DPR (C5: 35.38% Hit@1) narrows the gap, demonstrating that domain-specific training can make embedding retrieval competitive with LLM generation.

### 6.4 Error Analysis

We performed automated error classification on 200 randomly sampled incorrect predictions from each of O1 through O6, using GPT-4.1-mini to categorize errors into six types. Averaged across experiments:

**Table 4: Error Type Distribution (averaged across O1 through O6)**

| Error Type | O1 | O2 | O3 | O4 | O5 | O6 |
|---|---:|---:|---:|---:|---:|---:|
| Same-type, unrelated | 43.0% | 42.0% | 43.5% | 45.0% | 52.0% | 46.0% |
| Wrong type | 28.5% | 27.5% | 29.5% | 22.5% | 31.0% | 35.0% |
| Same-type, related | 24.5% | 25.5% | 22.5% | 24.0% | 13.5% | 17.0% |
| Partial match | 3.5% | 4.0% | 3.0% | 6.0% | 2.5% | 1.5% |
| Empty/hallucination | 0.5% | 1.0% | 1.5% | 2.5% | 1.0% | 0.0% |
| Ambiguous | 0.0% | 0.5% | 0.0% | 0.0% | 0.0% | 0.5% |

The dominant error mode across all models is **same-type, unrelated** (42-52%), where the model predicts an entity of the correct type but one that is semantically unrelated to the gold entity (e.g., predicting "Jack Johnson" when the gold is "Lou Ambers," both boxers). The second most common error is **wrong type** (22.5-35.0%), where the model predicts an entity of an entirely different category. **Same-type, related** errors (13.5-25.5%) represent cases where the prediction is semantically close to the gold (e.g., predicting "Okinawa" for "Iwo Jima"), suggesting partial cue recognition but insufficient disambiguation. Hallucinations and empty responses are rare (<2.5%), indicating that models reliably produce plausible entity names even when incorrect.

Notably, Llama 3.1 8B (O5, O6) shows a higher proportion of same-type, unrelated errors (52.0%, 46.0%) and a lower proportion of same-type, related errors (13.5%, 17.0%) compared to GPT models (O1, O2: 24.5%, 25.5%). This suggests that smaller models have weaker ability to narrow down candidates within a type using fine-grained contextual cues.

### A. Prompt Templates

#### A.1 Zero-Shot Prompt

**System message:**
```
You are an entity recognition expert. Given a text that implicitly
references a named entity without mentioning it, identify what entity
is being referenced.
```

**User message:**
```
What named entity is implicitly referenced in this text? The entity
is never mentioned by name.

Text: "{text}"

Think about the contextual cues (dates, places, events, people, roles)
and identify the specific named entity being referenced.

Answer with ONLY the entity name (canonical Wikipedia name), nothing else.
```

**Parameters:** temperature=0.0, max_tokens=100

#### A.2 Few-Shot Prompt

**System message:** (same as zero-shot)

**User message:**
```
What named entity is implicitly referenced in this text? The entity
is never mentioned by name.

Examples:
Text: "I remember that Sunday morning in December '41. We were listening
to the radio when the news broke about the attack on the naval base in
Hawaii. That's when everything changed."
Entity: Attack on Pearl Harbor

Text: "I enlisted right out of high school and went to boot camp in
San Diego. As an aircraft mechanic, I was sent to the Pacific."
Entity: United States Marine Corps

Text: "After the surrender, we flew into the main islands. I landed in
the bay and spent six months there for the occupation. The capital was
flattened by the B-29s."
Entity: Tokyo

Text: "In late 1941, I was set to ship out from San Francisco. A friend
ran up saying they're bombing the base in Hawaii."
Entity: Attack on Pearl Harbor

Text: "Growing up in that bustling metropolis with towering skyscrapers,
I was immersed in a vibrant culture."
Entity: New York City

Now identify the entity in this text:
Text: "{text}"

Answer with ONLY the entity name (canonical Wikipedia name), nothing else.
```

**Parameters:** temperature=0.0, max_tokens=100

#### A.3 Chain-of-Thought Prompt

**System message:**
```
You are an entity recognition expert. Think step by step.
```

**User message:**
```
What named entity is implicitly referenced in this text? The entity
is never mentioned by name.

Text: "{text}"

Think step by step:
1. What contextual cues are present? (dates, places, events, people, roles)
2. What type of entity do these cues suggest? (Person, Place, Organization, Event)
3. What specific named entity matches ALL these cues?

Reasoning: [your step-by-step analysis]
Entity: [canonical Wikipedia name]
```

**Parameters:** temperature=0.7, max_tokens=300

#### A.4 RAG Prompt

**User message (no system message):**
```
This text implicitly references a named entity without naming it.
Based on the contextual cues, which candidate is most likely?

Text: "{text}"

Candidates:
1. {candidate_1} - {description_1}
2. {candidate_2} - {description_2}
3. {candidate_3} - {description_3}
4. {candidate_4} - {description_4}
5. {candidate_5} - {description_5}

If none match well, suggest a better entity.
Answer: [number]. [entity name]
```

**Parameters:** temperature=0.7, max_tokens=50

### C. Example Predictions

We present five correct and five incorrect predictions from the best-performing model, O2 (GPT-4o, few-shot), to illustrate the task characteristics and failure modes.

#### C.1 Correct Predictions

**Example 1.**
- **Implicit text:** "I studied at a major public university in Northern California, where I was part of the Design Department. During my time there, I combined academic courses with art classes, focusing on three-dimensio[nal design]..."
- **Gold entity:** University of California, Berkeley
- **Prediction:** University of California, Berkeley

**Example 2.**
- **Implicit text:** "When I was in sixth grade, a teacher took us to the downtown area of a city in western Japan to see new buildings and illuminations. We visited a museum that later became the only building left standing after the bombing..."
- **Gold entity:** Hiroshima
- **Prediction:** Hiroshima

**Example 3.**
- **Implicit text:** "When I first came to America, I worked in a Pacific island territory for eight months on a sugar plantation. I was only 15 years old and worked under a Chinese boss for $18 a month. Although many Japa[nese]..."
- **Gold entity:** Hawaii
- **Prediction:** Hawaii

**Example 4.**
- **Implicit text:** "A city in western Kentucky was not directly discussed in detail, but it is one of the locations where the state's environmental programs have field offices..."
- **Gold entity:** Paducah, Kentucky
- **Prediction:** Paducah, Kentucky

**Example 5.**
- **Implicit text:** "This major Georgia city was not mentioned in my narrative, so I have no personal history or experiences related to that place."
- **Gold entity:** Atlanta
- **Prediction:** Atlanta

These correct examples demonstrate cases where geographic cues ("western Japan," "Pacific island territory," "western Kentucky," "Georgia") and contextual details (the atomic bombing museum, sugar plantations) are sufficiently distinctive for the model to identify the entity.

#### C.2 Incorrect Predictions

**Example 1.**
- **Implicit text:** "My great-grandfather left Lithuania in the early 1900s to escape oppression and seek a better life in America. He arrived before World War I and worked hard to establish himself, eventually sending for his family..."
- **Gold entity:** Solomon Goodman
- **Prediction:** Lithuanian Jews
- **Error type:** Wrong type. The model predicted a group/category rather than the specific individual being described.

**Example 2.**
- **Implicit text:** "I once kept a newspaper clipping of a write-up about a historic estate by a famous 19th-century author, but unfortunately, I have misplaced it. The author's writing gave me some insight into the estate..."
- **Gold entity:** Harriet Beecher Stowe
- **Prediction:** Monticello
- **Error type:** Wrong type. The model focused on the "historic estate" cue rather than the "famous 19th-century author" cue, predicting a place instead of a person.

**Example 3.**
- **Implicit text:** "The priest who taught me algebra and later became the bishop of a diocese in eastern Washington questioned my presence in his advanced algebra class because I lacked the necessary background..."
- **Gold entity:** Bishop of Spokane
- **Prediction:** West Point
- **Error type:** Wrong type. The model produced a completely unrelated entity, likely confusing the religious/military context.

**Example 4.**
- **Implicit text:** "Remote healthcare delivery became a critical part of how we provided care during the viral outbreak. Initially, we relied on phone calls, but within weeks, our organization quickly implemented video..."
- **Gold entity:** Telehealth
- **Prediction:** COVID-19 pandemic
- **Error type:** Same-type, related. The model identified the correct general domain but predicted the contextual event rather than the practice being described.

**Example 5.**
- **Implicit text:** "Jamilia Hanosh, also known as Jimmy, is my grandmother and the daughter of Fahima Hanosh and John Oirathy. She was born and raised in Michigan, part of a family that valued assimilation and education."
- **Gold entity:** Jamilia Hanosh
- **Prediction:** United States
- **Error type:** Wrong type. Despite the entity name appearing directly in the text (a pipeline failure where the name was not properly anonymized), the model still predicted incorrectly, choosing a broad geographic entity instead.

These errors illustrate the principal challenges of IER: distinguishing the referenced entity from related contextual entities (Example 4), resolving references to obscure individuals (Examples 1, 3), and focusing on the correct cue among multiple competing signals (Example 2). Example 5 also reveals an occasional pipeline artifact where entity anonymization fails to remove the name from the text.
